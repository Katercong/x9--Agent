> 格式：`[日期] [决策] [原因] [替代方案为何被否] [影响范围]`

## 2026-05-06

### D-001 数据库选型：SQLite 而非 Postgres / Notion / Airtable

- **原因**：本地 LAN，2 人团队，单文件易备份，零运维。SQLAlchemy 兼容 → 后期换 Postgres 一行连接串
- **否决项**：Postgres（要 server，运维成本高）/ Notion（API 速率限制，依赖外部）/ JSON+CSV（并发写冲突）
- **影响**：所有写入走 SQL，schema 演进通过 migrate_v*.py，备份就是复制 .db 文件

### D-002 单文件 HTML 前端 + 单一可信源

- **原因**：可分发性最强。整个项目压缩 zip 发给客户也能跑
- **否决项**：React + build step（构建工具链复杂）/ Streamlit（不够灵活）
- **影响**：前端用 vanilla JS，无 npm，仅靠浏览器 fetch + DOM

## 2026-05-07 早

### D-003 廖 / 张协作：API 是契约，不共享 SQLite 文件

- **原因**：廖的代码不读 db 文件，只调 `/api/v1/*`。张换实现不影响廖
- **否决项**：让廖直连 SQLite（每次张迁库就要协调）
- **影响**：所有写操作都有 audit log；schema 变化只要不破坏 API 就不用通知廖

### D-004 通用 `/api/v1/data/{resource}` + 自助建表

- **原因**：让廖能自己加新表（爬虫日志 / 视频时序 / 广告投放），不用每次找张加端点
- **否决项**：每个表写专门的 endpoint（永远写不完）
- **影响**：所有新表通过 `_meta_resource` 注册，generic CRUD 直接可用，廖建表不用通知张

### D-005 RBAC + 多用户独立 Key（专业人士提醒后改的）

- **原因**：单一全局 key 不能撤销 / 不能审计。每人独立 key + SHA-256 哈希存储 = 行业标准
- **否决项**：单 .api_key 文件（专业人士否决）/ JWT（过度设计）
- **影响**：admin / user / readonly 三级；token 服务端验证，前端只是携带

- **影响**：`llm_feature` 表，每个 feature 行可以填 provider_code + model；空时走全局 fallback

## 2026-05-08

### D-014 廖爬虫 lead 池：并行加表，不替换 creator 主表

- **背景**：廖给了一份 `create_x9_creator_database.py`，里面 `creators` 表带 60+ 字段（priority_score / fit_level / *_fit / *_score / recommendation_* / review_*），PK 是 VARCHAR(120)
- **决策**：方案 B（并行加表），新建 9 张表挂在 X9 库里，不动现有 `creator` 主表（INTEGER id）
- **否决项**：
  - 方案 A（完全替换 creator 表）：会断 101 条 outreach 流水的 creator_id INTEGER FK；会丢失 owner_bd / last_contact_date / current_status 等 BD 字段；语义上把"原始爬取池"和"已确认主表"硬塞一起
  - 方案 C（只开权限让廖手动建）：9 张表 + 79 tag seed 用 HTTP 一行一行戳太傻
- **影响**：
  - 廖立刻可写：`POST /api/v1/data/creator_leads/bulk`（slug 避让，底层表是 `creators`）
  - 张主表完整保留，66 创作者 + 101 outreach 不动
  - 后续要做一次 ETL 脚本：`creators` lead 池 → `creator` 主表（人工审核或自动 promote 看实际质量再定）
  - 命名容易混 → 术语表里写清楚 "creator (单数, INTEGER id, X9 主表) ≠ creators (复数, VARCHAR id, 廖 lead 池)"

### D-015 admin 可改任何内置表 schema（含张/廖）

- **原因**：v3.8.0 起廖（admin）需要给 `creator` / `outreach` 等加字段（爬虫数据多，字段会持续扩）。每次都让张手改 + 重启服务效率低
- **否决项**：
  - 只让特定用户名（如"liao"）能改 → 硬编码用户名违反 RBAC 设计
  - 加 `--allow-builtin` 显式标志 → 增加误操作风险但收益不高
- **影响**：`add_column` 撤掉 `is_dynamic` 卡口，仍走 `require_admin`。readonly/user 角色仍被拒。
- **风险**：admin 可能误改导致 schema 漂移 → 建议廖加列前到 #protocol 群里说一声 + CHANGELOG 公告
- **不开放的**：`DROP TABLE` / `DROP COLUMN` / `RENAME` 仍无 API 入口（要做就走 migrate_v* 脚本走流程）

### D-016 (2026-05-08 v3.8.2 修订)：DROP 端点 + key scope 都开放，但守住边界

廖需求单 P3 #9 + #10，张拍板都做。

- `DELETE /api/v1/tables/<name>?confirm=true` — **只允许 dynamic 表**，built-in (creator/outreach/product...) 仍 403。删 built-in 必须走 migrate_v*.py 重启。
- `DELETE /api/v1/tables/<name>/columns/<col>?confirm=true` — built-in / dynamic 都允许（admin 既然能 ADD 就该能 DROP），但拒绝删 PK / upsert_keys / json_cols / fk_lookup 涉及的列。
- `?confirm=true` 是对误操作的防护，不是安全控制。
- D-015 的 schema 改动开放给 admin 仍生效；D-016 是它的逻辑延续。

key scope（P3 #9）：
- 旧模型保留：未设 scopes 的 key 走 role 三档（admin/user/readonly），完全向后兼容
- 新模型叠加：设了 scopes 的 key 用 `assert_can(user, action, resource)` 做 resource 级匹配
- scope 字符串：`admin` | `admin:<pat>` | `write:<pat>` | `read:<pat>`，pattern 用 fnmatch
- 用例：给爬虫 worker 一把只能写 `tk_*` 资源的 key，丢了风险半径限定到 tk_* 数据；admin key（张/廖）保留全权限

## 2026-05-09

### D-017 creator 主表合并廖字段，方向是"加列+ETL"而不是"换主表"

- **背景**：v3.4.0 邀约话术生成器跑起来后，廖反馈他爬到的评分/推荐字段（priority_score / fit_level / evidence_text 等）在主表 `creator` 上都没有，话术生成器读不到。张提了"换主表"方案：rename `creator`→`creator_bak`，把 `tk_creators` 改名为 `creator`，BD 字段并入
- **决策**：否决换主表方案；改为给 `creator` ALTER ADD 50 列 + 一次性 ETL 把 `tk_creators` 的廖字段 UPSERT 进 `creator`
- **否决项（换主表）的硬伤**：
  - `outreach.creator_id` (INTEGER) 101 行 + `outbox.creator_id` 1 行 + 5 张表的 creator_id 列 — 两表 INTEGER PK 是各自 AUTOINCREMENT，1 号不是同一个人。改名后所有 FK 指向错误身份
  - BD 沉淀字段（owner_bd / current_status / first/last_contact_date）在 64 行重叠中是 BD 手填的，廖爬虫 130 条同名列大概率为空，硬覆盖会丢
  - 反向于 D-014（lead 池保持并行）的语义共识
- **影响**：
  - `creator` 现 132 行（66 BD 主表 + 66 廖独有 lead），FK 链完整
  - `tk_creators` 保留作廖原始落地表，廖继续往里写
  - 话术生成器 `build_prompt` 读廖字段 + evidence 片段
  - 下次需要合并：跑 `python scripts/migrate_v14.py`（idempotent）；廖加新列就把列名追加到 NEW_COLS

## 决策原则（持续遵守）

1. **加法不破坏，减法走流程** — 加字段/表/查询零通知；删/改字段先标 deprecated_note 至少 3 天
2. **API 是契约**，前端怎么改、张怎么换实现，廖代码不动
3. **写操作必须人类触发**，AI 永远只引导，不替用户提交
4. **服务端验证一切**，前端的 role 判断只是 UI 早失败提示
5. **KISS**：能用 SQL trigger 解决的不上 Python，能用 SQLite 单文件的不上 Postgres
