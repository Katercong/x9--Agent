# 张 ↔ 廖 协作约定

两人开发，一个数据库，一个 API。怎么不互相挡路？这份文档就是答案。

## 一、谁负责什么

| 区域 | 张 | 廖 |
|---|---|---|
| 数据库 schema (建表 / 加字段) | ✅ 主导 | 也能调 API 自助建（自己模块的表） |
| 数据填充 (产品 / 达人) | ✅ 通过前台界面录 | ✅ 通过 API 写入 (爬虫 / 自动化) |
| LLM Key + Provider 配置 | ✅ 在前台"设置"配 | 不接触 Key，只调 `/api/v1/llm/complete` |
| 前端 UI (管理界面) | ✅ 主导 | 不动 |
| 业务规则 (命名查询 / 状态机) | ✅ 主导，告诉廖 | 调用 |
| 爬虫 / 自动私信 / 视频跟踪 / 广告跟踪 | 不动 | ✅ 主导 |
| 自动化定时任务 | 不动 | ✅ 主导 |
| 任务总表对应 (需求确认表) | 1.x / 2.x / 3.1 / 3.2 / 5.x | 2.2.2 / 3.1.2 / 3.2.3-4 / 3.4.4 / 3.5.4 / 4.x |

## 二、热改 vs 冷改 — 你做改动是否影响廖

| 改动类型 | 重启吗 | 廖那边 | 怎么做 |
|---|---|---|---|
| 加产品/达人/SKU 数据 | ❌ | 无感 | 前台 UI 或 `/api/v1/data/*/bulk` |
| 改 SKU 价格/卖点/达人状态 | ❌ | 无感 | 前台 UI |
| 配 LLM Key / 切 Provider | ❌ | 无感 | 前台 "设置" |
| 自助建新表 / 加新列 | ❌ | 无感 | `POST /api/v1/tables` |
| **加新命名查询** | ❌ | 无感 | `POST /api/v1/queries`（DB 存，热加载） |
| 修改已有命名查询 | ❌ | 无感 | `PUT /api/v1/queries/{name}` |
| 改 Python 代码（v1.py / llm.py / registry.py） | ✅ ~1-2s | **几乎无感**（现代 HTTP client 自动重连） | 直接保存文件，uvicorn `--reload` 自动重启 |
| 改 schema.sql 加新内置表 | ✅ ~1-2s | 同上 | 改完保存 |

`run.bat` 默认开了 `--reload`，监听 `app/` 和 `scripts/` 目录。文件保存即重启，停顿 1-2 秒。HTTP 客户端只要带最基本的 retry 就完全无感。

## 三、API 是契约，不是代码

廖的代码**永远只跟 `http://192.168.1.168:18765/api/v1/*` 讲话**，不读数据库文件、不依赖张的 Python 代码。这意味着：

- 张内部怎么改实现都行（SQLite → Postgres、字段加索引、UI 重写……），只要 API URL 和响应字段没变，廖代码不动
- 廖要新字段 / 新查询，跟张说，张加一个**就行**，不需要联调
- 调试时双方各自跑：廖在自己机器上跑爬虫脚本，张在自己机器上看数据库，互不干扰

## 三、什么时候要打招呼

### 不需要打招呼（绝大多数情况）

- 加新 SKU、新达人、新建联事件
- 给已有表加新字段
- 加新表（`POST /api/v1/tables`）
- 加新命名查询
- 改前端 UI
- 调 LLM Provider / Key

### 需要打招呼（少数情况）

| 场景 | 提前几天 | 怎么做 |
|---|---|---|
| 重命名字段 | ≥1 天 | 在 `_meta_resource` 表对应行写 `deprecated_note='use new_name instead'`，老字段保留 1 周 |
| 删字段 | ≥3 天 | 先 deprecate，3 天后真删；CHANGELOG 写 `-` |
| 删 endpoint | ≥3 天 | 同上 |
| 重命名 resource | ≥1 天 | 同字段重命名 |
| 改 API 响应结构 | ≥3 天 | 极罕见，需要专门讨论 |
| 重置服务端 X-API-Key | 当时通知 | 廖那边 1 行 env var 改下 |

## 四、廖如何感知"张那边改了什么"

3 个机制，自上而下从轻到重：

### 1. 实时 schema 发现（最轻）
```bash
GET /api/v1/resources              # 当前所有 resource + 字段定义
GET /api/v1/resources/{name}       # 某一个 resource 的完整 schema
GET /api/v1/queries                # 当前所有命名查询
```

### 2. 版本 + 时间戳（中）
```bash
GET /api/v1/version
```
返回 `server_version` 和 `changelog_last_modified_ts`。廖可以每天 / 每运行一次 poll 一次，看到变了就扫一眼 CHANGELOG。

### 3. CHANGELOG.md（人读）
[`CHANGELOG.md`](CHANGELOG.md) — 每次张改东西都加一行。廖每周看一眼。

## 五、廖如何不影响张

- **不要改公共 schema**（creator / product / outreach 这些）。要改先跟张说
- **可以自由建自己模块的表**（`POST /api/v1/tables`），如 `ad_campaigns` / `video_metrics_history` / `email_outbox` 这种。命名建议加前缀，如 `ad_*` / `video_*` / `auto_*`
- **不要直接连 SQLite 文件**，永远走 API。否则张迁库到 Postgres 那天廖就要重写
- **遇到 500/502 不要重试不停**，先看 `detail`，找张排查

## 六、典型协作场景

### 场景 A：廖要做"广告投放跟踪"
1. 廖调 `POST /api/v1/tables` 建 `ad_campaigns` 表（参数自己定）
2. 廖往里写数据：`POST /api/v1/data/ad_campaigns/bulk`
3. 张在前端"设置 → 自定义 Provider"看到了这表（resource 列表里多了一项），但**不动它**
4. 张要看广告数据 → 直接调 `GET /api/v1/data/ad_campaigns?status=running` 或在前端做个 tab
5. 全程零联调

### 场景 B：张觉得 `creator.notes` 字段不够用，想拆成 `notes_zh` 和 `notes_en`
1. 张在 CHANGELOG 加一行 `~ creator.notes 即将拆分为 notes_zh + notes_en (5/15 生效)`
2. 在 `_meta_resource.deprecated_note` 标 `notes` 字段为 deprecated
3. ALTER TABLE 加 `notes_zh` / `notes_en` 两个新列
4. 写迁移脚本把现有 `notes` 内容拷过去
5. 通知廖：5/15 之前请把代码里 `notes` 改成 `notes_zh`
6. 5/15 删掉旧 `notes` 列

### 场景 C：廖发现需要一个"按 SKU 统计已发视频数量"的查询
1. 廖跟张说："能加个查询 X 吗？"
2. 张在 `app/registry.py` 里加一行 `_q("videos_per_sku", ...)`
3. 廖立即可用 `GET /api/v1/queries/videos_per_sku`

## 七、Key 丢失 / 重置 / 紧急恢复

### 服务端只存 SHA-256 哈希，明文丢了找不回

跟所有正经系统一样：服务端 `api_key.key_hash` 是不可逆哈希，反推不出原文。
所以 "丢 key 了" 的解法 **永远是签发新的 + 撤销旧的**，不存在"找回原 token"。

### 三种重置场景

| 场景 | 怎么办 |
|---|---|
| 廖丢了 token，但张能登 | 张登录 → 设置 → 用户管理 → 廖 → 查看 Keys → 撤销旧 key → 签发新 key → 发给廖 |
| 张丢了 token，但 `.api_key` 文件还在 | 用 `.api_key` 里那串重新登录前台 |
| 张丢了 token 而且 `.api_key` 也删了 / 整个 localStorage 清空 | 命令行兜底：在 server 这台机器上跑 `reset_key.bat zhang`，会撤销所有旧 key 并签发一把新 admin key 到控制台 |

### CLI 重置工具

`F:\Claude_Project\Database\reset_key.bat` —— 双击即可使用：

```
reset_key.bat                    列出所有用户及 key 数量
reset_key.bat liao               给廖签发新 key（撤销他原有的所有 key）
reset_key.bat liao --add         追加一把新 key，不撤销原有的（多设备场景）
reset_key.bat zhang              紧急重置 admin（你自己锁死自己时的最后一招）
```

新 token 直接打到控制台，**只显示一次**，复制走。

### 为什么 CLI 是兜底而不是用 API

API 自身需要登录，"我登不上了所以要重置" 形成循环依赖。CLI 跑在 server 这台机器上，访问权限来自文件系统读权限 = 你已经是这台机器的管理员。

### 安全建议

- 重置完张 admin key 后，**删掉项目根目录 `.api_key` 文件**（明文存在文件系统是个弱点）
- 廖那边收到 token 后存到密码管理器（1Password / Bitwarden / 系统钥匙串），不要留在聊天记录里
- 半年轮换一次 key（设置 → 用户管理 → 撤销旧的 → 签发新的）
- 实习生离职时立即撤销其 key；用户可以保留（活作 inactive）便于查审计

## 九、最差情况：张电脑挂了 / WiFi 断了

廖那边的爬虫 / 自动化脚本会调不通 API。这种情况：

1. 脚本应该有重试（指数退避）
2. 短期断线（< 30 分钟）不算事
3. 长期断线（> 1 小时）廖直接微信问张
4. 真的张电脑长时间不能开机，张可以备份 `database.db` + 整个 `Database/` 目录给廖临时跑

后续如果团队 ≥ 5 人 / 频繁出门，就走云部署（README 里写了迁云方案，schema 不动，连接串改一行）。

## 十、给廖的最佳实践

```python
# 推荐封装
from x9_db_client import X9DB    # 廖自己写的 client (按 docs/廖_API使用指南.md 第九节模板)

db = X9DB()

# 1. 先发现，再编码
schema = db.get_resource("creators")   # GET /api/v1/resources/creators
print(schema["columns"])                # 看清楚有哪些字段
# 编码时只用列出来的字段；张以后加字段不会破坏

# 2. 写代码时容错
try:
    res = db.bulk("creators", items)
    if res.get("errors"):
        # 单条错不影响整体，记日志即可
        log.warning("partial errors:", res["errors"])
except requests.RequestException:
    # 网络失败：重试
    time.sleep(retry_delay)

# 3. 自己模块的表自己建，加前缀
db.create_table("scraper_run_log", columns=[...])   # 我的爬虫日志，不污染主 schema

# 4. 定期 poll version (可选)
v = requests.get(f"{BASE}/api/v1/version").json()
if v["changelog_last_modified_ts"] > self.last_seen:
    print("张那边改了 schema，去 docs/CHANGELOG.md 看看")
```
