## 1. 服务启停

### 1.1 启动

双击 `run.bat`。浏览器自动打开 `http://localhost:18765/`，登录后进入主界面。

第一次启动会跑全部迁移脚本（v2 / v3 / v4 / v5），无需手动。之后每次启动都会跑一遍（idempotent）。

### 1.2 停止

**正确做法：双击 `stop.bat`** —— 它会找到监听 18765 的 python 进程，连同子进程一起干净杀掉。

**不要做**：直接点 cmd 窗口右上角 ×。
- Windows 上 uvicorn `--reload` 会派生主进程 + worker 子进程
- 强关 cmd 主进程死了，worker 子进程留下来占着端口
- 下次启动会报 "Port already in use"
- 那时只能再 `stop.bat` 或重启电脑

### 1.3 重启

双击 `restart.bat`（= stop.bat + run.bat 一气呵成）。

什么时候要重启：
- 改了 schema.sql 或运行了大数据导入
- 切换了 LAN（IP 变了）
- 误操作让服务卡住

什么时候**不**用重启：
- 改 Python 代码（自动热重启）
- 改 SQL 数据 / SKU / 达人 / 命名查询 / LLM 配置
- 加新表 / 加新列

### 1.4 自动热重启（dev 模式）

`run.bat` 默认开了 `--reload`，监听 `app/` 和 `scripts/` 下任何 .py 改动。

工作流：
```
1. 改 app/v1.py 加个新接口，按 Ctrl+S 保存
2. cmd 窗口里 uvicorn 自动打印 "Reloading..."
3. 1-2 秒后新代码生效
4. 廖那边的脚本如果有 retry 装饰器，几乎零感知
```

如果改完代码 cmd 窗口报语法错误：服务会**保持旧版本继续运行**，错误会在窗口红字显示。修正语法后再保存即可。

### 1.5 端口冲突

`run.bat` 启动前会预检 18765 是否被占。被占时不会强启，会清晰报错：

```
[ERROR] Port 18765 is already in use.
Edit run.bat and change "set PORT=18765" to a free port (e.g. 18766, 8910, 9100).
```

改端口步骤：
1. 用记事本打开 `run.bat`，把 `set PORT=18765` 改成 18766（或任何空闲端口）
2. 同步改 `stop.bat` 里的 `set PORT=` 那行
3. 通知廖把他代码里的 BASE URL 改成新端口
4. 重新双击 `run.bat`

### 1.6 启动失败排查

| 报错 | 原因 | 解决 |
|---|---|---|
| `Python 3.11 not found` | Python 路径不对 | 改 `run.bat` 第 6 行 `set PY=` 指向你的 python.exe |
| `Port already in use` | 上次没干净关 | `stop.bat` 后再启 |
| `database.db is locked` | 另一个进程在写 | 等几秒再启；或 `stop.bat` |
| `ModuleNotFoundError: fastapi` | 依赖缺失 | `python -m pip install fastapi uvicorn pandas openpyxl pdfplumber python-docx requests` |
| 启动了但浏览器打不开 | 防火墙拦了 / IP 错 | 浏览器手动访问 `http://127.0.0.1:18765/` 试本机；不行再看防火墙 |

## 2. 用户和权限

### 2.1 角色总览

| 角色 | 能做 | 不能做 | 适合谁 |
|---|---|---|---|
| `admin` | 全权：所有写操作 + 用户管理 + 改 schema + 配 LLM Key + 删数据 | — | 张 / 廖 / 长期合作的核心开发 |
| `user` | 写数据 / 加表 / 加列 / 加命名查询 / 调 LLM `/complete` | 用户管理 / 删数据 / 配 LLM Provider Key | 实习生 / 外包开发 |
| `readonly` | 只读 + 调 LLM `/complete` | 任何写操作 | 看数据的同事 / 领导 / 财务 / 数据分析师 |

> **核心原则**：从最低权限开始（principle of least privilege）。能给 readonly 的不给 user，能给 user 的不给 admin。

### 2.2 新增管理员（交接 / 领导要权限）⭐

**❌ 错误做法**：把自己的 key 复制粘贴给对方

**✅ 正确做法**（5 步）：

1. 前台登录（用你自己的 admin key）
2. 设置 → 用户和 Key 管理 → 展开 "+ 创建新用户"
3. 填写：
   - username: `lingdao`（或拼音简写，只能字母数字下划线）
   - 显示名: `领导张三`（中文也行）
   - 角色: 选 `admin`
   - 备注: `2026-05-07 临时给行政查看用` 之类的
   - 点 "创建"
4. 在用户列表里找到这个新用户，点 "+ 签发新 Key" → 弹窗里那串 token 复制
5. 把 token 通过**安全渠道**发给对方（推荐：1Password 共享 / Bitwarden Send / 当面），**不要发微信群 / 邮件**

对方收到后，访问 `http://192.168.1.168:18765/`，登录框粘贴这串 token 即可。

> 💡 给领导建议用 `readonly` 角色，他们一般只是想看数据，不应该有写权限。除非确实需要批准/否决某些操作。

### 2.3 新增普通用户（实习生）

同 2.2，但角色选 `user`。这样他能写数据、加表，但碰不到用户管理和敏感操作。

### 2.4 新增只读用户（管理者查看）

同 2.2，但角色选 `readonly`。这是最常见的"邀请同事看数据"场景。

### 2.5 用户离职流程 ⭐

**当天要做**：
1. 前台 → 用户管理 → 找到那个人
2. 点 "查看 Keys" → 一把把记下 key_id → 输入 key_id 撤销（也可以从命令行：`reset_key.bat <username>` 一键撤销所有 key 但不签新的 —— 但目前 reset_key 默认会签新的，要避免就别用，去前台撤就行）
3. 改他的角色为 `readonly`（保留账号，所有历史 audit_log 还能查到他做了什么）
4. 点 "停用"（active = 0）—— 即使将来 key 真泄露也无法登录

**1 周后**（确认没问题）可以彻底删除：
- 点 "删除" → 二次确认 → 用户和所有 key 一并消失

> ⚠️ 系统强制保留至少 1 个活跃 admin，删不掉最后一个。
> ⚠️ 你不能删自己。

### 2.6 工作交接流程 ⭐

比离职稍复杂，因为要把"管理员身份"交出去：

```
1. 接手人入职第一天，按 2.2 创建他的 admin 账号
2. 让他登录前台，调一个写操作（如改某个达人的 notes 字段）确认 key 工作
3. 把所有运维相关文件位置告诉他：
   - 项目目录：F:\Claude_Project\Database\
   - .local_keys_backup.txt
   - 这份操作手册
   - 协作约定.md（与廖的协作规则）
4. 让他熟悉 1 周（看看用户列表 / audit_log / 数据分布）
5. 第 2 周开始让他独立操作（你在场协助）
6. 完全交接当天：
   a. 删掉 .local_keys_backup.txt（或重新生成只含他+廖的版本）
   b. 撤销你自己的 key
   c. 把你的用户改为 readonly + 停用（保留账号便于追溯历史）
```

> 💡 不要在交接当天删自己 —— 万一接手人 key 也丢了就锁死了。先停用，1 个月后再删。

### 2.7 修改用户角色（晋升 / 降级）

前台 → 用户管理 → 那个用户 → "改角色" 按钮 → 输入 admin / user / readonly。

CLI 等价：
```python
# 直接改 DB 也行（紧急时）
import sqlite3
con = sqlite3.connect(r'F:\Claude_Project\Database\database.db')
con.execute("UPDATE api_user SET role='user' WHERE username='intern1'")
con.commit()
```

### 2.8 临时停用用户

用户出差 / 休假 / 借调，但不想删账号：
- 前台 → "停用" 按钮（active = 0）
- 他的 key **立即失效**（所有请求 401）
- 想恢复 → 同一按钮变成 "启用"

### 2.9 永久删除用户

前台 → "删除" 按钮 → 二次确认。
后果：用户记录消失，他名下所有 key 同步消失（CASCADE）。`audit_log` 里他过去的操作记录**保留**（用 username 字符串记录的）。

### 2.10 一人多设备（多个有效 Key）

廖在公司电脑 + 家里电脑都要跑爬虫脚本。两种做法：

**A. 共用一把 key**（简单）：
- 直接把同一串 token 拷到两台机器
- 风险：一台被入侵两台都暴露

**B. 一人多 key**（推荐）：
- 张作为 admin 给廖签 2 把 key（可以加 description 区分："廖-公司"、"廖-家"）
- CLI: `reset_key.bat liao --add` 追加一把不撤销旧的
- 任意一台丢了单独撤销那一把，另一台不受影响

### 2.11 撤销可疑 Key

发现某个 key 被误传到了不该的人手里：
- 前台 → 用户管理 → 那个用户 → "查看 Keys" → 输入要撤销的 key_id
- 撤销后**立即生效**（下一次请求就 401）
- 同时记得给他发一把新的

### 2.12 定期 Key 轮换

建议节奏：
- admin key：3-6 个月
- user key：6-12 个月
- readonly key：12 个月或离职时

操作：
1. 给目标用户 "+ 签发新 Key"
2. 把新 key 发给他
3. 等他确认新 key 工作（在前台登录或调一次接口）
4. 撤销他的旧 key

## 3. Key 丢失与恢复

> 服务端只存 SHA-256 哈希，**明文不可还原**。所有恢复方案都是"撤旧 + 签新"，不存在"找回原 token"。

### 3.1 别人丢了 Key（你作为 admin）

最常见情况，前台一键搞定：
1. 前台 → 用户管理 → 那个用户
2. "+ 签发新 Key" → 复制弹窗 token
3. 通过安全渠道发给他
4. 顺手 "查看 Keys" → 撤销他的旧 key（防止后续误用）

### 3.2 自己丢了 token（但 `.local_keys_backup.txt` 还在）

直接打开 `F:\Claude_Project\Database\.local_keys_backup.txt`，复制你那串重新登录。

### 3.3 自己丢了 token 且备份也没了

CLI 兜底：双击 `reset_key.bat zhang`（或换成你的 username）：

```
[OK] Done.
NEW API KEY for user 'zhang' (role: admin):

  4Q3xT...新 token...

⚠ This is the ONLY time you'll see this token. Copy it now.
```

复制走，重新登录。

### 3.4 双 admin 都锁死了

只要有一个人能开机访问 `F:\Claude_Project\Database\` 目录：
- `reset_key.bat zhang` 重置一把
- 拿这把登录前台
- 给廖也重置 + 重发

如果你们俩都不在这台电脑前 —— 这是**为啥不要把数据库放在某一个人电脑**的根本原因。后期上云能解决（schema 不动，只换连接串）。

### 3.5 数据库损坏

见 [§8 灾难恢复](#8-灾难恢复)。

## 4. 协作流程（与廖）

> 完整规则见 [协作约定.md](协作约定.md)。下面是**操作动作**速查。

### 4.1 谁做什么（默认分工）

| 张 | 廖 |
|---|---|
| schema 设计、字段定义 | 自己模块的表（爬虫日志 / 视频时序）自助建 |
| 公共表数据维护（产品 / 达人主信息） | 数据写入（爬虫产出、自动邮件、视频跟踪） |
| LLM Provider Key 管理 | 调 `/api/v1/llm/complete` |
| 业务命名查询 | 调用查询 |
| 用户管理（含给廖签 key） | 不动 |
| 前端 UI | 不动 |

### 4.2 哪些改动**需要**通知廖

只在做这几件事时通知（占改动的 5%）：

- 删字段 → 提前 3 天 + 先标 deprecated_note
- 改字段名 → 提前 1-3 天 + CHANGELOG 注明 旧名 → 新名
- 删 endpoint → 同删字段
- 改服务器端口 / IP → 当天通知
- 重置/撤销廖的 key → 当天通知

### 4.3 哪些改动**不用**通知

90% 的改动都不用：
- 加 SKU / 达人 / 建联事件
- 加新字段 / 新表 / 新查询 / 新 LLM Provider
- 改 UI / 改业务规则 / 改 Python 代码
- 重启服务（廖那边 retry 装饰器自动恢复）

### 4.4 张加新查询的流程

> 全程不重启服务。

```bash
# 用 admin key 调 API
curl -X POST http://localhost:18765/api/v1/queries \
  -H "X-API-Key: <张的key>" -H "Content-Type: application/json" \
  --data '{"name": "videos_per_sku", "description": "每个SKU有多少视频", "sql": "SELECT ...", "params": [["limit","int",50]]}'
```

或在 OpenAPI 调试页 (`/docs`) 里点点。

廖立即就能用 `GET /api/v1/queries/videos_per_sku`，不用通知。

### 4.5 廖加新表的流程

廖自己调 `POST /api/v1/tables`，张那边的前端会自动识别新 resource。
廖建表后**不用通知**张 —— 除非他想让张在前端做个 tab 展示。

### 4.6 张要加字段（非破坏性）

直接调 `POST /api/v1/tables/{name}/columns`，或者改 `schema.sql` 重启。
新字段自动出现在 `GET /api/v1/resources`，廖的代码不动也照常工作（generic CRUD 自动忽略他不认识的字段）。

### 4.7 张要重命名字段（破坏性）⚠️

走流程：
1. 在 `_meta_resource` 表对应行写 `deprecated_note='use new_name (effective YYYY-MM-DD)'`
2. ALTER TABLE 加 `new_name` 字段（不删旧的）
3. 写迁移脚本把 `old_name` 内容拷过去
4. CHANGELOG 加 `~ rename old_name → new_name`
5. 通知廖（微信文字+ CHANGELOG 截图）
6. 等 1 周（让廖代码改完）
7. 真删旧字段，CHANGELOG 加 `-`

## 5. 数据维护

### 5.1 新一周的统计表来了

把新文件放进 `F:\实习生\C达人建联\` 后双击 `reimport.bat`。
导入脚本对已有记录是 upsert（不会重复导），新达人 / 新事件会进库。

### 5.2 新增 SKU（产品上新）

两种做法：
- **少量**：前台 → 产品 → "+ 新增 SKU"，逐字段填
- **批量**：编辑 `scripts/import_products.py` 顶部的 `SKU_META` 加新条目，重跑 `python scripts/import_products.py`

### 5.3 重新整理产品图片（实习生加了新图）

```
python scripts/import_images.py
```

幂等，已有 product_image 记录不会重复，只补新图。

### 5.4 导出 xlsx 给非技术同学

```
python scripts/export_xlsx.py
```

输出在 `exports/`：
- 产品总表.xlsx
- 达人总表.xlsx
- 建联流水.xlsx

xlsx 是**只读镜像**，发出去给同事自己改不影响数据库。

### 5.5 数据库备份

**每周做一次**：
1. 双击 `stop.bat` 停服
2. 把整个 `F:\Claude_Project\Database\database.db` 文件复制到 OneDrive / 移动硬盘
3. 双击 `run.bat` 重启

或者用脚本（不停机版本）：
```python
# scripts/backup.py
import sqlite3, datetime, shutil
src = r'F:\Claude_Project\Database\database.db'
dst = rf'D:\backups\database_{datetime.date.today()}.db'
con = sqlite3.connect(src)
con.execute(f"VACUUM INTO ?", (dst,))   # SQLite 内置热备份
con.close()
print(f"backed up to {dst}")
```

### 5.6 迁移到云

后期团队人数多了或要异地协作时：
1. 在云服务器（阿里云轻量 30/月）装 Python 3.11 + 依赖
2. 拷整个 `F:\Claude_Project\Database\` 目录上去
3. 改 `app/main.py` 把 SQLite 换成 Postgres（连接串一行改）
4. 防火墙开 18765
5. 廖那边 BASE URL 从 `192.168.1.168:18765` 改成云 URL
6. **API 协议完全不变**，他代码不用改

## 6. AI 项目管理员（内置咨询助手）

前台导航多了一个 "🤖 AI 助手" tab。它是项目内置的**只读 AI 顾问**：加载了所有运维文档（操作手册 / 协作约定 / API 指南 / schema），可以回答你或同事关于这个系统的任何疑问。

### 6.1 启用前提

需要先在"设置 → LLM 配置中心"激活一个 Provider 并配 Key（参见 §6 LLM 配置 旧节，现移至 §6.5）。
未启用时，AI 助手 tab 顶部会显示"⚠ 暂未启用，去设置页配置"。

### 6.2 它能做什么

- 解答问题："怎么给新人加管理员？" / "schema 的 creator.tier 是什么意思？" / "廖那边报 401 怎么排查？"
- 给步骤指导：每个回答都引用具体的手册章节 + 操作动作
- 解读错误：贴 cmd 报错给它，它告诉你可能的原因
- **嵌入"操作入口"按钮**：回答里直接给可点击的蓝色按钮，点了带你到对应页面/查询/弹窗，省去自己摸索路径

### 6.2.1 它的两种入口

| 入口 | 在哪 | 适合 |
|---|---|---|
| **悬浮 FAB**（右下角 🤖 圆按钮） | 所有 tab 都能看到 | 在做某事时随时问"这个怎么操作"，不打断当前页面 |
| **完整 tab**（顶部导航 "🤖 AI 助手"） | 顶部导航 | 长对话 / 排查问题 / 给同事演示 |

两个入口**共享同一份对话历史**（localStorage 同步），随便切。

### 6.2.2 操作入口按钮（彩蛋）

AI 助手回答里会出现蓝色按钮，比如：

> 改 SKU 价格的步骤：
> 1. 🧭 [打开产品页] ← 这是按钮，点击直接切到产品 tab
> 2. 找到要改的 SKU，点击
> ...

支持的按钮类型：
- 🧭 切换 tab（产品 / 达人 / 建联 / 设置 ...）
- 🔍 直接打开某个 SKU / 达人的编辑窗
- ⚡ 运行命名查询并弹窗显示结果
- 🎯 切到目标 tab 并应用筛选条件（如"筛 A 级达人"）

按钮点击后会带你过去，但**写操作还是你自己点保存** —— AI 没有写权限。

### 6.3 它不能做什么（设计上的硬限制）

- ❌ **不能修改任何数据 / 文件 / schema** —— 即使你是 admin，它也只引导你自己点
- ❌ **不会回显完整 token / key** —— 文档里出现也只露前 4 位
- ❌ **不给业务决策**（要不要给某人 admin / 这个达人能不能合作） —— 这是人的判断
- ❌ **不会编造**手册里没写的内容，会明说"这个手册里没写"

### 6.4 推荐用法场景

| 场景 | 让 AI 助手干 |
|---|---|
| 新人入职第一天 | 先逛一圈 AI 助手 tab，问几个常见问题熟悉系统 |
| 你忘了某个操作 | 不用翻手册，直接问"我要 ××，怎么做？" |
| 廖报错你拿不准 | 把错误截图描述贴进去，让它分析 |
| 领导问你"这字段啥意思" | 让 AI 答（比你查 schema.md 快） |
| 想加新功能但不确定影响 | 问"我想加 ××，要走哪些流程？" |

### 6.5 LLM 配置中心

> AI 助手、未来话术生成、内容分析等所有 AI 功能都从这里统一选 Provider 和 Key。

#### 6.5.1 添加新 Provider（如 Moonshot Kimi）

前台 → 设置 → "+ 添加自定义 Provider"
- code: `moonshot`
- 显示名: `Kimi`
- 协议: `OpenAI 兼容`
- Base URL: `https://api.moonshot.cn/v1`
- 默认模型: `moonshot-v1-8k`
- 创建后回到 Provider 列表，粘贴 API Key → 保存 → 测试连通 → 设为活跃

#### 6.5.2 切换 Provider

设置 → 找到目标 Provider → "设为活跃"。
廖那边调 `/api/v1/llm/complete` 不传 `provider` 时自动用新的活跃 Provider。

#### 6.5.3 撤销 Provider 的 Key（怀疑泄露 / 公司换 OpenAI 账号）

设置 → 那个 Provider → "清除 Key"。
该 Provider 立即变成"未设 Key"状态，活跃也会被取消。
重新粘贴新 Key + 保存 + 设为活跃即可。

#### 6.5.4 测试连通

设置 → 任一 Provider → "测试连通"（发 8 token 的 ping，~1 秒）。结果显示在 Provider 卡片右上角。

## 7. 突发故障排查

### 7.1 廖那边报 502 Bad Gateway

LLM 上游报错。流程：
1. 张前台 → 设置 → LLM Provider → 看 Active Provider 的 "测试连通"
2. 失败 → 看 detail：是 401（key 错）还是 429（限流）还是网络断
3. 据此处理：401 → 换 key；429 → 等几分钟或换 Provider；网络断 → 看你的网络

### 7.2 廖那边报 401 Unauthorized

廖的 X-API-Key 头没带或写错或被撤销了：
1. 你前台 → 用户管理 → 廖 → 查看 Keys → 看他活跃的 key 还在不在
2. 不在 → 重新签发一把发给他
3. 在 → 让廖确认他代码里粘的是不是这一串

### 7.3 廖那边报 403 Forbidden

廖的角色不够：
- 写数据要 user/admin（readonly 不行）
- 改 schema 要 admin
- 删数据要 admin

去用户管理调他角色，或者让他换调用方式。

### 7.4 廖那边报 404 Not Found

URL 写错或资源不存在。让他先调：
```
GET /api/v1/resources       看所有资源名
GET /api/v1/queries         看所有查询名
```

### 7.5 自己前台显示数据异常 / 加载失败

按 F12 打开浏览器控制台 → Network 标签 → 看红色请求。
常见：
- 401 → token 过期或被撤销，登出重新登录
- 500 → 服务器报错，看 cmd 窗口红字 traceback

### 7.6 改了 .py 代码服务没自动重启

cmd 窗口看有没有 "Reloading..." 字样。如果没有：
- `--reload` 监听的是 `app/` 和 `scripts/`，改其他位置的 .py 不触发
- 文件没真正保存（查看修改时间）
- uvicorn 报语法错误了 —— 修语法
- 实在不行 → `restart.bat`

### 7.7 端口被占（不是上次的我）

`netstat -ano | findstr "18765"` 看是哪个 PID
`tasklist /fi "PID eq <那个PID>"` 看是什么程序
不是你的就改 `run.bat` 换端口。

### 7.8 日志在哪看

cmd 窗口本身就是日志。如果 cmd 关了找不到日志：
- audit_log 表（DB 里）有所有写操作
- 服务异常错误：cmd 窗口红字 traceback

要持久化日志：把 `run.bat` 最后一行改为
```
"%PY%" -m uvicorn app.main:app --host 0.0.0.0 --port %PORT% --reload --reload-dir app --reload-dir scripts > logs\server.log 2>&1
```

## 8. 灾难恢复

### 8.1 database.db 文件损坏

如果服务起不来 + cmd 报 `database disk image is malformed`：
1. 立即 `stop.bat`
2. 复制损坏的 `database.db` 到 `database.db.broken`（保留现场）
3. 用最近备份替换 `database.db`
4. `run.bat` 重启验证
5. 损失：备份点之后的修改丢失（看你备份频率）

修复尝试（高级）：
```
sqlite3 database.db.broken ".dump" | sqlite3 database.db.recovered
```
如果 dump 能跑完，replace `database.db` 用 `database.db.recovered`。

### 8.2 服务器机器整个挂了

如果 `database.db` 文件还能读出来：
- 拷到任何装了 Python 3.11 的机器
- 把整个 `F:\Claude_Project\Database\` 目录复制过去
- 改 `run.bat` 里 `set PY=` 指向新机器的 python.exe
- 双击 `run.bat`
- 让廖把 BASE URL 切到新 IP

### 8.3 双管理员都丢 key 且 .local_keys_backup.txt 也没了

只要还能开机访问 `F:\Claude_Project\Database\`：
- `reset_key.bat zhang` → 控制台拿到新 key
- 然后给廖也 `reset_key.bat liao`

### 8.4 开机失败 + 文件系统也坏了

只能寄望于上次备份的 `database.db`。
**所以**：每周备份到 OneDrive（5.5）。
