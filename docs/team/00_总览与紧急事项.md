## 在拉新人进来之前,必须先处理的 4 件事

下面这几件按顺序处理,**全部处理完再发邀请**,否则会出大问题。

### 🚨 第 1 件:密钥泄漏修复(最高优先级)

经审计,以下敏感信息**已经进了 git 历史并被推到 GitHub**:

| 密钥 | 出现在哪些提交 | 怎么处理 |
|---|---|---|
| Gmail OAuth `client_secret`(`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`) | `a612622`, `ddcd8fa`, `5abf7fc`, `bec1a16`, `ff1de04`, `3169234` | **立即吊销** + 重新生成 |
| PostgreSQL 密码(`x9_local_dev_2026`) | 同上 + 多处 | **立即修改**,所有 `.env` 同步更新 |

**具体操作:**

1. **吊销 Gmail OAuth 凭据**
   - 打开 [Google Cloud Console → Credentials](https://console.cloud.google.com/apis/credentials)
   - 找到 project `x9-mail-oauth` 下的 OAuth Client ID `564835011967-h4dnh7dt5habohbrgfd7m249h53tg6r8`
   - **Reset Secret**(或直接删掉重建)
   - 新的 `client_secret` 写进本机的 `.env.shared`,**不要**写进任何 git 跟踪的文件

2. **修改数据库密码**
   ```powershell
   docker exec -it x9-postgres psql -U x9 -d x9db
   # 在 psql 里:
   ALTER USER x9 WITH PASSWORD '<新的强密码>';
   \q
   ```
   然后更新 `.env.shared` 里的 `X9_PG_PASSWORD` 和 `LOCAL_DB_URL`,重启所有服务。

3. **关于 git 历史**:重写 git 历史(`git filter-repo` / `BFG`)会导致所有人重新克隆,而且历史里的密钥即使删了,GitHub 缓存和 fork 也可能还在。**推荐的更现实做法是:认定密钥已泄漏 → 吊销 → 不重写历史**。GitHub 仓库即使是 private,被授权过的所有人(包括之前给过权限的)都可能看到。

4. **未来防止再次发生**:见 **02 — GitHub 配置清单** 里"开启 Secret Scanning"那一节,以及 **01 — 协作规范** 的 commit 前检查清单。

### 🧹 第 2 件:清理当前工作区

当前状态(`git status`):

- 你在 `codex/datafix` 分支上,有 **487 个未提交修改**
- 仓库里还有 8 个 `claude/...` 命名的本地分支(疑似 AI 助手历史会话残留)
- 重复目录:`desktop/` 和 `x9_creator_desktop_system/` 看起来是同一个东西,README 里也提到了"合并"但似乎没清理干净
- 旧的 SQLite 数据库文件还在 `core/database.db`、`core/database.db.bak_v14` 等位置(虽然不在 git 里)

**建议步骤:**

```powershell
# 1) 先把当前修改全部 stash 起来(不要丢)
git stash push -u -m "pre-team-cleanup-2026-05-20"

# 2) 切回 main,跟远程对齐
git checkout main
git pull origin main

# 3) 删掉本地 claude/* 旧分支(确认远程已没有再删)
git branch | findstr "claude/" | ForEach-Object { git branch -D $_.Trim() }

# 4) 从 stash 里把真正要保留的改动一点点挑出来,作为正式的 feature 分支提交
git stash list
# 然后 git stash show -p stash@{0} 看一下,决定怎么处理
```

如果 487 个改动里大部分是"不该提交"的本地产物(比如 `dist/`、`logs/`、`.bak`、生成的快照),先补 `.gitignore`,然后再来挑剩下的真改动。

### 📋 第 3 件:补齐 `.env.shared.example`

新人没法直接用你机器上的 `.env.shared`(里面有真密钥),需要一个"占位符版本"提交到 git 里作为模板。

模板已经准备在 `docs/team/templates/.env.shared.example`,直接复制到仓库根目录即可:

```powershell
copy docs\team\templates\.env.shared.example .env.shared.example
git add .env.shared.example
git commit -m "docs: add env template for team onboarding"
```

确保 `.gitignore` 里继续保留 `.env.shared`(实际密钥不入库),但 `.env.shared.example` **要**入库。

### 🛡️ 第 4 件:开启 GitHub 分支保护

参考 **02 — GitHub 仓库配置清单**,至少做完这 3 件最低限度:

- `main` 分支开启保护(禁止 force push、禁止直接 push、要求 PR)
- PR 至少 1 人 review 通过才能合并
- 开启 Secret Scanning + Push Protection(免费)

做完这 4 件,就可以拉新人进来了。

## 你"既是开发者又是服务器维护者"的特殊处理

你提到自己也要在这台 Windows 上继续开发。这台机器同时承担着两个角色,必须分开管理,否则你的开发改动会让"上线版本"飘忽不定。

**推荐方案 — 两个克隆 + 端口分离:**

```
F:\X9_AI_system\        ← 生产克隆(只跟 main 同步,跑给团队/用户用的服务)
                        ports: postgres 15432, core 18765, desktop 8000
                        
F:\X9_dev\X9_AI_system\ ← 你的开发克隆(在 feature 分支上写代码,可以随便折腾)
                        ports: postgres 25432, core 28765, desktop 18000
```

完整细节见 **03 — 本地开发环境** 的"维护者(你)的特殊配置"小节。核心规则:

- **生产克隆里只做 `git pull origin main`**,从不直接写代码、从不切分支
- 写代码全在 dev 克隆里,提 PR → review 通过 → merge 到 main → 在生产克隆 `git pull` → 重启服务
- 两套数据库完全隔离,dev 操作不会污染生产数据
