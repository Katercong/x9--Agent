# 01 — 团队协作开发规范

> 全员必读。这是日常开发的"交通规则"。
> 设计原则:**对 Git 新手友好**、流程尽量少、出错容易回滚。

---

## 1. 分支模型(GitHub Flow,简化版)

我们用最简单的模型,不用 Git Flow:

```
                ┌────────────────────────────┐
main 分支  ────►│  始终是"线上可用"的版本  │ 只能通过 PR 合并进来
                └────────────────────────────┘
                       ▲             ▲
                       │ PR + Review │ PR + Review
                       │             │
              feat/xxx ┘   fix/yyy  ┘    ← 每个人在自己的 feature 分支上写代码
```

**就这两层。** 没有 `develop`、没有 `release`、没有 `hotfix` 分支。

### 1.1 分支命名规范

| 前缀 | 用途 | 例子 |
|---|---|---|
| `feat/` | 新功能 | `feat/creator-batch-tagging` |
| `fix/` | 修 bug | `fix/email-outreach-duplicate-send` |
| `chore/` | 杂活(依赖升级、清理) | `chore/upgrade-fastapi-0.115` |
| `docs/` | 只改文档 | `docs/api-audit-may-2026` |
| `refactor/` | 重构(行为不变) | `refactor/extract-llm-router` |
| `data/` | 数据迁移/修复 | `data/dedupe-creators-by-email` |

**规则:**
- 一律小写,单词之间用 `-` 连接
- 名字要描述清楚做了什么,不要 `feat/update`、`fix/bug`
- 一个分支只做一件事(后面 PR 才好 review)
- 你自己写的分支,不需要带名字前缀。但如果一个长期分支需要标注 owner,用 `feat/rocky-batch-tagging`

### 1.2 分支生命周期

```
新功能开始
  │
  ├─ git checkout main
  ├─ git pull origin main              ← 永远从最新的 main 开始
  ├─ git checkout -b feat/xxx
  │
  │     ...写代码、commit...
  │
  ├─ git push -u origin feat/xxx
  ├─ 去 GitHub 提 Pull Request
  ├─ 等 review(可以一边等一边继续修)
  ├─ 通过 → Squash and Merge 到 main
  │
  └─ 删除远程和本地分支(GitHub 提供按钮 / `git branch -D feat/xxx`)
```

**不要做的事:**
- ❌ 直接在 `main` 上写代码(GitHub 也会拦)
- ❌ 用一个分支做多个无关的功能("顺便修了个 bug" → 那个 bug 应该单独开分支)
- ❌ 把已经合进 main 的分支留着不删
- ❌ 在别人的 feature 分支上直接 push(除非那人邀请你 co-author)

## 2. Commit 规范(Conventional Commits 简化版)

格式:

```
<类型>: <一句话说明做了什么>

[可选的更详细说明,空一行]
```

**类型**(和分支前缀对应):

`feat` / `fix` / `chore` / `docs` / `refactor` / `data` / `test` / `style`

### 好例子

```
feat: 为创作者列表添加按 GMV 排序

按用户反馈,在 /api/v1/creators 加 ?sort=gmv_desc 参数,
默认仍然按 updated_at 排序避免影响现有调用方。
```

```
fix: 修复 Gmail 外联在收件人含 + 号时 OAuth 报错

原因:URL 编码时把 + 当成空格处理。改用 quote(safe="@.+") 修。
```

```
data: 去重 creators 表中 email 相同的记录(保留最新)
```

### 坏例子

```
update         ← 改了啥?
fix bug        ← 哪个 bug?
WIP            ← 提交前 squash 掉
asdfasdf       ← 别这样
更新了一下代码  ← 改了什么?
```

### Commit 频率

- 一个"逻辑改动"一个 commit。比如"加新字段 + 写迁移脚本 + 改用 UI"可以拆成 3 个 commit
- 提交前 `git diff --staged` 自己看一遍,确认没多余的东西(`.env`、日志、个人配置、AI 助手生成的临时文件)
- 一个 commit 改一千行?基本上是不对的,拆

### Commit 前必做的 3 个检查

```powershell
# 1) 看清楚自己要提交什么
git status
git diff --staged

# 2) 确认没把密钥/数据库/大文件夹之类带进来
git diff --staged --stat | findstr /R "\.env \.db \.bak password secret key"

# 3) 跑一下基本检查(见 03 文档里的 lint 配置)
# python: ruff check . && ruff format --check .
# node:   npm run lint
```

## 3. Pull Request 流程

### 3.1 提 PR 之前

- [ ] 自己跑过一遍能跑(本地启动 / 单元测试 / 主要功能点过一次)
- [ ] 分支跟最新 main 同步过(rebase 或 merge main 进来,见下方"冲突处理")
- [ ] commit 是干净的(没有 `WIP`、没有"哦这个不该提交"那种)
- [ ] PR 标题用和 commit 一样的风格:`feat: 添加按 GMV 排序`

### 3.2 PR 描述应该写什么

模板已经放在 `.github/pull_request_template.md`,提 PR 时 GitHub 会自动填。最少要回答这三个问题:

1. **改了什么**(一句话)
2. **为什么改**(对应哪个 issue / 谁提的需求)
3. **怎么验证**(reviewer 如何确认这个 PR 是对的)

### 3.3 PR 的大小

- 理想:**< 400 行 diff**,15 分钟能 review 完
- 上限:超过 800 行 diff 的 PR 必须有 reviewer 同意才能合(否则拆)
- 唯一例外:依赖升级 / 自动生成的代码 / 数据迁移文件,在 PR 描述里说明

### 3.4 合并方式

- **默认 Squash and Merge**:多个 commit 合并成一个 commit 进 main,history 整洁
- 如果一个 PR 里有多个独立的有意义的 commit(罕见),可以 `Rebase and Merge`
- **从不使用** `Create a Merge Commit`(会让 main 历史变成蜘蛛网)

### 3.5 紧急修复(hotfix)的特殊流程

线上炸了,不能等 review:

1. 还是开 `fix/xxx` 分支(不要在 main 上写)
2. 改完直接推上去、提 PR
3. **找一个人快速过一眼**(任何在线的人都行,< 10 分钟)
4. 合并 → 在生产克隆 pull → 重启
5. 事后写一个简短的"事故复盘"贴在 issue / Slack

不允许"线上太急,我直接在服务器改文件了"——这会让 git 和实际文件不一致,后面所有人都会被坑。

## 4. Code Review 规范

### 4.1 Reviewer 的责任

你是 reviewer 时,看以下几点:

| 维度 | 看什么 |
|---|---|
| **正确性** | 这个改动会真的实现它说要做的事吗?边界条件想到了吗?(空列表、None、超长输入、并发) |
| **安全** | 有没有把密钥、token、内部 URL 写死进代码?用户输入有没有被 SQL/Shell/HTML 注入的可能? |
| **数据库** | 有没有 schema 变更?有没有写迁移?会不会锁大表? |
| **可读** | 半年后我重新看这段代码,能看懂吗?变量名清晰吗? |
| **测试** | 改的逻辑核心点有没有 test 覆盖?(不要求 100%,但关键路径要有) |
| **影响范围** | 这个改动会不会影响别的人的工作?需要通知谁? |

### 4.2 评论的方式

**好评论:**
- "这里如果 `creators` 为空列表,line 47 会抛 IndexError,建议加个判断"
- "建议把这段抽成 `_normalize_email()` 函数,后面 batch import 也会用到"
- "👍 这个用 generator 改写比之前清爽多了"

**坏评论:**
- "代码风格不太行"(具体哪里?)
- "你应该用 X 而不是 Y"(为什么?有性能/可读性/安全的原因吗?)
- 在每行都写一句"OK"(只看了表面)

### 4.3 谁来 review

- 默认由 GitHub 自动分配(基于 `.github/CODEOWNERS`,见 02 文档)
- 至少 **1 个 approve** 才能合
- 涉及数据库迁移 / 安全 / 部署脚本的 PR,**必须 Rocky 或指定的另一个人 approve**
- 自己的 PR 不能自己 approve(GitHub 也会拦)

### 4.4 SLA

- 工作日上午提的 PR,**当天下午前要么 review、要么明确说"我今天没空,找 XX"**
- 不要让 PR 在 review 队列里躺超过 2 个工作日
- 如果 reviewer 给了改动意见,**作者要么改、要么明确回复"我认为不需要改,理由是…"**

## 5. Issue 管理

我们用 GitHub Issues,**所有非琐碎的工作都要有 issue**。

### 5.1 什么时候开 issue

- 发现 bug → 开 issue,贴上 `bug` label
- 有新需求 → 开 issue,贴上 `feature` label
- 技术债 / 重构想法 → 开 issue,贴上 `tech-debt`
- "我只是临时改个错别字" → 不用开 issue,直接 PR

### 5.2 Issue 模板

模板放在 `.github/ISSUE_TEMPLATE/`,有两种:
- `bug.md` — bug 报告(必填:复现步骤、期望、实际)
- `feature.md` — 需求提议(必填:为什么、想要什么、可以接受的方案)

### 5.3 Issue 状态用 Label

| Label | 含义 |
|---|---|
| `bug` | 已确认的 bug |
| `feature` | 新功能 |
| `tech-debt` | 重构/优化 |
| `priority:high` | 1 周内必须处理 |
| `priority:low` | 有空再说 |
| `blocked` | 等别的东西,无法推进 |
| `good-first-issue` | 适合新人接手 |

### 5.4 PR 和 Issue 的关联

在 PR 描述里写 `Closes #123`,合并 PR 时 GitHub 会自动关闭那个 issue。

## 6. 冲突处理 — 写给 Git 新手

最让新人怕的就是 merge conflict。这里讲一下怎么稳妥地处理。

### 6.1 提 PR 前发现 main 有了新提交

```powershell
# 你在 feat/xxx 上,main 已经被别人推了新东西
# 推荐:把 main 的最新改动合并进你的分支

git checkout main
git pull origin main
git checkout feat/xxx
git merge main
```

- 如果没冲突:`git push` 完事
- 如果有冲突:VS Code 会高亮显示。每个 `<<<<<<< / =======/ >>>>>>>` 块都要手动决定保留哪边。改完:
  ```powershell
  git add <冲突文件>
  git commit                   # commit message 默认就行
  git push
  ```

### 6.2 我把分支搞乱了怎么办

**铁律 #1:不要 `git reset --hard` 或 `git push --force`**,除非你完全清楚自己在做什么。

发现自己搞乱了:
1. **不要慌**,你的代码 99% 还在
2. `git stash` 把所有未提交的改动暂存
3. `git status` 看清楚现在在哪个分支、状态是什么
4. **在 Slack/群里说一声**,让有 Git 经验的人帮你看一眼

兜底大招:
```powershell
# 备份当前分支
git branch backup-feat-xxx-2026-05-20

# 然后从 main 重新拉一个干净的分支
git checkout main
git pull
git checkout -b feat/xxx-redo

# 手动 cherry-pick 你想要的 commit
git cherry-pick <commit-hash>
```

### 6.3 永远不要 force push 到 main

`main` 已经在 GitHub 配置里禁了 force push,但其他分支没禁。**只有自己一个人在用的 feature 分支**可以 force push,**有别人在协作的分支不可以**。

## 7. 安全红线(没得商量)

以下任何一条出现,都属于严重事故,需要立即处理:

1. **把密钥/token/密码 commit 进 git** —— 即使马上删,git 历史里就有了。预防:用 `.env.shared.example` 作为模板;`.env.shared` 永远在 `.gitignore` 里;开启 GitHub Secret Scanning。
2. **把生产数据库的 dump 文件 commit 进 git** —— 包含用户数据
3. **把客户/达人个人信息明文写进代码或 commit message**
4. **在生产环境用 `git checkout` 跨分支跳来跳去** —— 永远 pull main 后重启,见 03 文档

如果出了 1-2,**立刻 Slack 通知 Rocky**,然后吊销/重置那个凭据。**不要试图自己悄悄 force push 删掉**,这会留下更多痕迹。

## 8. 新人入职清单

新成员加入团队第一天,跟着这个清单走:

- [ ] 拿到 GitHub 仓库邀请,接受
- [ ] 加入团队 Slack / 钉钉,自我介绍
- [ ] 读完 `docs/team/00`-`04` 这五份文档(约 1 小时)
- [ ] 按 `docs/team/03` 配置本地开发环境
- [ ] 跑通 `start_all.ps1`(或 dev 版本),能在浏览器打开 `:18765`
- [ ] 找一个 `good-first-issue` 标签的 issue,练习走一遍完整的 feature 分支 → PR → review → merge 流程
- [ ] 把第一个 PR 合并后,这个人就算"入职完成"

## 9. 沟通约定

- **PR 评论 / Issue 讨论**:用文字,留底,异步,默认所有人都能搜到
- **Slack / 钉钉群**:讨论怎么做、咨询、闲聊
- **当面 / 视频**:需要画图、需要快速对齐方向的时候
- **不要**用私聊讨论代码方案 —— 别人搜不到,后面会重复回答同样的问题
